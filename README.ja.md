> [!WARNING]
> インストールするとバックグラウンドデーモンが**自動的に実行**されます。その一部は毎日自動的に Claude を呼び出すため、放置していてもトークン(コスト)が消費されます。
>
> さらに、このハーネスは**トークン使用量が多いシステム**です。オーケストレーターが一つのリクエストを専門エージェントのチームへ分解するため、リクエスト一つが複数のサブエージェント呼び出しへと広がり、トークン消費が単一エージェントの数倍に達することがあります。

## 何なのか

Glass Atrium は、Claude Code CLI を、オーケストレーションされ、観測可能で、自己改善されるマルチエージェントシステムへと変える設定・ツールのレイヤーです。これは新しいエージェントフレームワークでは**なく**、Claude Code を置き換えるものでも**ありません** — `~/.claude/` にインストールされ、あなたがすでに使っている CLI の上で動作します。

デフォルトの状態で、一つの一貫したシステムとしてデプロイされる五つを追加します:

- 単一の汎用プロンプトの代わりに、**能力ベースでルーティングされる専門エージェントの部隊**(スタック別の開発者、QA、企画、リサーチ、デザイン、セキュリティ、ウィキ、メタ);
- どのエージェントがどの規則をロードするかをマトリクスで定義する**階層化された規則システム**;
- すべてのツール境界でその規則(シークレット・危険コマンド・予算・成果物)を*機械的に*強制する**ライフサイクルフックパイプライン**;
- すべてのコスト・結果・エージェントイベントを一箇所に現す**リアルタイムモニタリングダッシュボード**(Atrium Monitor);
- 繰り返される失敗を修正するためにエージェント自身の指示を書き直す**自己改善ループ**。

## なぜ作ったのか

単一の `CLAUDE.md` プロンプトはスケールしません。一つのモデルに対して、React の専門家であり*同時に*データベースの専門家であり*同時に*セキュリティレビュアーであり*同時に*企画者であれと要求した瞬間、指示は互いに衝突し、重要な規則は膨大なテキストの山の中で希釈され、モデルは負荷がかかると静かにその規則から逸脱し始めます。

次に思い浮かぶ明白なステップ — エージェントに作業を互いに引き渡させること — は、一つの問題をより悪い問題へと変えてしまいます。エージェント間の**ハンドオフ**システムでは、制御権がエージェント A から B へ、さらに C へと渡り、各エージェントは自分の規則を自ら強制せねばならず、観測可能性は連鎖全体にわたって断片化し、途中にある一つの誤作動エージェントが歯止めなく伝播します。責任を負う主体が誰もいません。

Glass Atrium は、統制を手放さずに専門化の利点を得るために作られました。すべての専門エージェントは継続的な中央監督のもとで実行され、重要なすべての規則はプロンプトではなくハーネスが直接強制し、システムは何が起きたかを記録して — 自動的に — そこから学習します。

## 核心アイデア: ハンドオフ連鎖ではなく一つのオーケストレーター

Glass Atrium は **Manager Pattern(中央集権マネージャーパターン)** の上に築かれています。一つのオーケストレーター — メインの Claude Code セッション — が恒久的な**戦略的制御プレーン**です。その唯一の役割は、あなたの意図をサブタスクへと*分解*し、各タスクを適切な専門エージェントへ*委任*し、結果を*統合*することにあります。自ら直接コードを書いたり成果物を作り出したりすることは決してありません。

逆のモデルである **Handoff Pattern(ピア間のエージェント対エージェント制御移譲)** は**明示的にサポートしていません** — まさにその点が核心です。

Glass Atrium において、オーケストレーターはすべての委任境界で制御権を保持します:

- **誰が何をするかを把握しています。** 各サブタスクはキーワードではなく能力でルーティングされます(キーワード/エイリアスのマッチングは禁止)。
- **すべての結果を検証します。** 各エージェントが構造化された完了記録を返し、オーケストレーターが意図対結果の整合性を確認します。
- **中断・再割り当て・エスカレーションができます。** 失敗したサブエージェントは失敗復旧ループ(リトライ → フォールバック → デバッガーエスカレーション)を作動させます。
- **境界があります。** スポーン深度・並行性の上限とエージェント別の予算が強制され、いかなる委任も暴走できません。

この統制は**単に手続き的なものではなく機械的です。** 規則をプロンプトに書いておいて守られることを願うのにとどまらず、ライフサイクルフックがすべてのツール呼び出しを横取りし、モニターがすべてのイベントを観測し、結果記録によってすべての委任を追跡・監査できます。その組み合わせ — 中央ルーティング **+** フックに裏打ちされた強制 **+** 完全な観測可能性 **+** 閉じた学習ループ — こそが、統制されないハンドオフシステムに対する核心的な差別化点です。

## 動作の仕組み、最初から最後まで

一つのリクエストは、次のようにシステムを経ていきます:

1. **あなたがオーケストレーター(メインセッション)に**何かを依頼します — バグ修正、機能企画、レポート作成。
2. **オーケストレーターが調査・分解**したうえで、能力レジストリを参照し、専門エージェントのチームと実行順序を構成します。
3. **各サブタスクを委任**します — すべての委任は、アクション実行の*前に* **PreToolUse フック**(危険コマンドの遮断・シークレットスキャン・範囲逸脱の表示・計画検証ゲート)を経ます。
4. **専門エージェントが作業を遂行**します — **SubagentStart フック**が注入した自身の範囲規則と、強制されたスポーン/ターン/ツール予算の中で。
5. **結果が記録されます。** 各エージェントが `[COMPLETION]` ブロックを放出し、**PostToolUse フック**がこれを結果記録として捕捉します。
6. **オーケストレーターが統合**します — 検証された結果を一つの回答へ、失敗時には復旧ループを実行します。
7. **すべてが観測可能です。** すべてのコスト・結果・エージェントイベントが PostgreSQL と Atrium Monitor へリアルタイムにストリーミングされます。
8. **システムは自ら改善します。** バックグラウンドデーモンが結果記録と是正シグナルを読み取り、エージェントの指示を自動的にパッチします — プロンプトを一行も編集しなくても。

ハーネスを部品から組み立てるわけではありません。`./glass-atrium` の対話メニューから **Install** を一度選べば、システム全体が一度の検証された実行で構築され(設定、シンボリックリンクファーム、フック連結、データベース、モニタービルド、ヘルスゲート)、インストール後にバックグラウンドデーモンが自動的に起動します。実行可能なコマンドは[クイックスタート](#クイックスタート)にあります。

## 大きな仕事は慎重に、小さな仕事は素早く

このシステムは、**どの段階を経るかを作業の大きさに合わせて自ら決めます。** あなたが「慎重に段階を踏め」といちいち指示しなくても済みます。

仕事が大きく複雑であれば、システムは慎重な順序を自動的に踏みます。平たく言えば、**計画を立てて → その計画が妥当かを点検し → 実際に作り → 正しく動くか試し → すべて完了したか確認して締めくくる**という流れです。逆に小さな仕事は、この手順を飛ばして素早く処理します。どちらへ進むかは、作業の大きさと複雑さを見てシステムが自ら判断します。

ここで「ワークフロー」という言葉が出てくることがありますが、これは**上記の段階をどのような方式で実行するか**を指す言葉です。別途「ワークフローで進めろ」と指定すると、同じ段階を人間が一つずつ指示する代わりに、自動化されたエンジンが一括でまとめて処理します。重要な点は、点検・試験などの安全装置と経る段階は**どちらでも同じ**ということです。変わるのは「どう実行するか(自動エンジンが一括で処理するか、一段階ずつ処理するか)」だけです。

> 一行要約: **何を経るかは作業の大きさが自動的に決め、どう実行するかだけが選択肢です。**

## 内部の構成要素

- **専門エージェントの部隊** — キーワードではなく能力レジストリ(`agent-registry.json`)でルーティングされます(スタック別の開発者、QA、企画・リサーチ・レポート、デザイン・オーディオ、セキュリティ、ウィキ、メタ)。
- **階層化された規則システム** — グローバル憲章(`agents/GLOBAL_RULES.md`)、コアの横断規則(`rules/`)、範囲別の規則(`scoped/`)を、明示的なコンプライアンスマトリクスで束ねます。
- **ライフサイクルフックパイプライン** — すべてのツール境界でシークレット・危険コマンド・予算・結果記録を機械的に強制するフックスクリプト群(`hooks/`)。
- **自己改善ループ** — autoagent デーモン(`autoagent/`)が、蓄積された結果記録と是正シグナルをエージェント指示のパッチへと変え、安全なものだけを git ベースのロールバックとともに自動適用します。
- **Atrium Monitor** — Fastify 5 + Prisma 7 + React 18 ベースの 10 画面リアルタイムダッシュボード(`http://127.0.0.1:7842`)。
- **編集可能なモデル + 予算割り当て** — モニターの **Models & budgets** 画面が、ドメイン別のモデルと呼び出しごとの USD ハードキャップを、設定ファイルを編集せずに割り当てます。
- **ライブアーキテクチャマップ** — モニターの System map 画面が、維持管理される 7 個の Mermaid ダイアグラムを、ライブのデーモン状態とともにレンダリングします。
- **ウィキ知識ストア** — 原本 → キュレーションされたノートのパイプラインと、SQLite BM25 全文検索インデックスを備えた LLM 専用ストア(`wiki/`)。リサーチエージェントの Web 調査結果がここに蓄積され、新しい調査・分析を始める前にまず参照して既存の知識を再利用します。
- **内部エージェントスキル** — エージェントとオーケストレーターが自動呼び出しする、漸進的開示の `SKILL.md` パッケージ([スキル](#スキル-内部の品質レイヤー)参照)。
- **ファイル単位のシンボリックリンクファームインストール** — ユーザー所有のファイルと共存できるよう、ディレクトリではなくファイル単位で `~/.claude/<rel>` → `~/.glass-atrium/<rel>` のシンボリックリンクを冪等に生成します。
- **8 個のバックグラウンドデーモン** — `com.glass-atrium.*` の launchd ジョブ(3 個の keepalive: monitor・autoagent・wiki · 5 個のスケジュール: ログローテーション・PG バックアップ・autoagent cycle・wiki compile・日次再起動)、すべて `config.toml` に宣言。
- **リポジトリにシークレット資料ゼロ** — peer-auth 専用の PostgreSQL(パスワードなし)、シークレットスキャンフック、リリースゲートの PII スキャン。

## スキル: 内部の品質レイヤー

Glass Atrium のスキルは、ユーザーが呼び出すコマンドでは**ありません** — セッション開始時にグローバルにロードされ、Claude が当面の作業に応じて自動的に有効化する、内部の品質・ガバナンスレイヤーです。コード/デザインの規約、安全不変条件(「鉄則」)、運用・検証ゲート、ウィキ・Web ツールを扱い、一部はモニターのシグナル(System map のドリフトバッジ、Models & budgets の保存など)に反応して発動します。**どのスキルが動いているかを知る必要はありません** — ハーネスが「ただ動く」ことの一部として選び、呼び出します。

## クイックスタート

**前提要件**: macOS · Claude Code CLI。それ以外のすべては、インストール TUI が同意を得て自動的に検出・インストールします。

```sh
curl -fsSL https://github.com/bettep-dev/glass-atrium/raw/main/install.sh | bash
```

対話メニューが開いたら **Install** を選んでください — インストール後に依存関係とデーモンが自動的に設定・起動し、ダッシュボードは `http://127.0.0.1:7842` で応答します。

> クローンしたフォルダはそのままの場所に残してください。Atrium はディレクトリを丸ごと差し替えるのではなく、ファイル単位のシンボリックリンクファームとして動作します。Claude の設定ディレクトリ内のファイルが、クローンした `~/.glass-atrium` の中にある実体ファイルを指すシンボリックリンクになるため、あなた自身のファイルと衝突せずに共存できます。実体ファイルはこのプロジェクトフォルダの中にあり、インストールされたシンボリックリンクはそこを指しているので、このフォルダを移動したり削除したりするとリンクが壊れます。

### アンインストール

メニューから **Uninstall** を選んでください。インストールされたシンボリックリンクを取り除き、GA データベースを削除して、Atrium を既存の Claude システムからきれいに、そして完全に切り離します。あなた自身のファイルはそのまま残り、Atrium の痕跡は何も残りません。データベースは設計上バックアップを取らずに削除されますが、再インストール時には新しいデータベースが作り直されます。

### Atrium Monitor のドキュメント作成方法

ドキュメント・レポート・要約・リファレンスを依頼すると **intel-reporter**(レポート・要約・リファレンス担当)へ、企画・要件定義・作業分解を依頼すると **intel-planner**(企画・計画担当)へ委任されます。

どの形式で出力されるかは**リクエストの文言が決めます**:

- デフォルトはモニター上に**隠された「エージェント専用」のトークン最適化記録**です — 内容の形に応じて md・yaml・json・txt の中から選びます。後で再び参照するための内部用記録です。
- 「**HTML で**」「**Web ドキュメントとして**」「**チーム共有用に**」のように共有・閲覧の意図を明確に示すと、人間が閲覧・共有できる **HTML ドキュメント**が作成されます(ダークテーマ・ダイアグラム・表を備えた単一ファイル)。

どちらの形式も Atrium Monitor の**ドキュメント管理**画面に表示されます。

例:

- 「今回の変更の振り返りレポートとしてまとめて」 → エージェント専用記録(デフォルト)。
- 「チーム共有用 HTML レポートにして」 → HTML ドキュメント。

ドキュメントを明示的に依頼しなくても、記録しておく価値があると判断されれば、エージェント専用記録が残されることがあります。

### 開発(DEV)エージェントの追加

「○○をする開発エージェントを追加して」のように依頼すると、オーケストレーターはすぐに新しいエージェントを作らず、まず**既存エージェントを拡張(EXTEND)するか、新規生成(CREATE)するか**を判断します。**デフォルトは拡張であり、生成はゲートを通過したときのみ**許可されます。

生成が許可されるには、三つが**すべて独立して**満たされる必要があります:

- **異なる成果物の種類** — 作り出すファイルが構造的に区別されます。
- **異なる決定ドメイン** — 正しい判断に必要な専門性が重なりません。
- **移転不可能な品質判断** — 一方の品質レビューを他方の専門性だけでは代替できません。

これに加えて、提案されたエージェントのドメインが**既存エージェントと一定比率以上重なると生成がブロックされます** — 大きく重なるなら、新規作成のシグナルではなく既存エージェントを拡張するシグナルだからです(重複防止)。

進行中には**ユーザー承認の一時停止が二度**あります:

1. **生成 vs 拡張の決定承認** — オーケストレーターが判断根拠と推奨案を提示すると、あなたが確定します。
2. **登録(コミット)承認** — 実際にエージェントファイルとレジストリへ記録する直前に、もう一度確認します。

本文(システムプロンプト)は **meta-prompt-engineer** が作成し、登録自体は専用 CLI が処理します。要約すると、あなたは「追加して」と依頼するだけで済みます — 拡張と新規のどちらが適切かはオーケストレーターが判断して提案し、新しく作る際には承認を求めます。
