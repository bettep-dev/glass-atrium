オーケストレーターが一つのリクエストを受け取り、専門エージェントのチームへ分解・委任したうえで、その結果をまとめ上げます。あらゆるコスト・結果・エージェントイベントは、ガラス越しに中をのぞき込むようにリアルタイムのモニターダッシュボードへ映し出され、システムは繰り返す失敗から自らの指示を修正していきます — プロンプトに手を入れる必要はありません。

> [!WARNING]
> インストールするとバックグラウンドデーモンが**自動で起動**します。その一部は毎日 Claude を自動で呼び出すため、放置していてもトークン(コスト)を消費します。
>
> しかも、このハーネスは**トークンを大量に使うシステム**です。オーケストレーターが一つのリクエストを専門エージェントのチームへ切り分けるため、リクエスト一つが複数のサブエージェント呼び出しへと広がり、トークン消費が単一エージェントの数倍に達することもあります。
>
> ただし、このトークンは Anthropic API に直接接続して従量課金される類のものではありません。このハーネスはすでに契約している **Claude Code CLI の上で**動作するため、個別の API キーや従量課金なしに、その契約プランに含まれるトークンをそのまま使う仕組みです。

## 何なのか

Glass Atrium は、Claude Code CLI を、オーケストレーションされ、観測可能で、自己改善するマルチエージェントシステムへと変える設定・ツールのレイヤーです。新しいエージェントフレームワークでは**なく**、Claude Code を置き換えるものでも**ありません** — `~/.claude/` にインストールされ、あなたがすでに使っている CLI の上で動きます。

「ガラスのアトリウム」という名前は、このシステムを捉えるためのレンズでもあります。AI を賢い一人の人間と見なすのではなく、夜でも中が透けて見えたまま 24 時間動き続ける一棟のガラスの建物 — その中で大勢の専門家が働きながらも、あらゆる業務が一人の統括者を通る会社として捉えるのです。隠しごとのないガラス、そしてすべての道が集まる中央の統括者 — この二つこそが、名前に込めたアイデンティティです。

そのうえで、デフォルトのインストールだけで一つの一貫したシステムへと束ねられる五つを加えます:

- 単一の汎用プロンプトですべてをこなすのではなく、**能力を基準にルーティングされる専門エージェントの部隊**(スタック別の開発者に加え、QA・企画・リサーチ・デザイン・セキュリティ・ウィキ・メタ);
- どのエージェントがどの規則を読み込むかをマトリクスで定義する**階層化された規則システム**;
- すべてのツール境界で、その規則(シークレット・危険コマンド・予算・成果物)を*機械的に*強制する**ライフサイクルフックパイプライン**;
- あらゆるコスト・結果・エージェントイベントを一箇所に集めて見せる**リアルタイムモニタリングダッシュボード**(Atrium Monitor);
- 繰り返す失敗を正すために、エージェント自身が自らの指示を書き換える**自己改善ループ**。

## なぜ作ったのか

たった一つの `CLAUDE.md` プロンプトでは、規模に耐えられません。一つのモデルに向かって、React の専門家であると*同時に*データベースの専門家であり、*同時に*セキュリティレビュアーであり、*同時に*企画者であれと求めた途端、指示同士がぶつかり合います。肝心の重要な規則は膨大なテキストに埋もれて薄まり、負荷がかかるとモデルはいつのまにかその規則から外れていきます。

だからこそ、仕事は分けるべきなのです — 一人の天才よりも複数の専門家の協働が勝りますし、分ければそれぞれがきれいな記憶領域(コンテキスト)を持って自分の仕事に集中できます。

とはいえ、その次の一手 — エージェント同士に作業を引き渡させること — は、かえって問題を大きくします。エージェント間の**ハンドオフ**システムでは、制御権が A から B へ、B からさらに C へと渡り歩きます。各エージェントは自分の規則を自力で守らねばならず、観測可能性はばらばらに散り、一つの誤作動エージェントが歯止めなく波及しても、最後まで責任を負う主体が誰もいません。

Glass Atrium は、統制を手放すことなく専門化の恩恵を得るために作りました — その答えが中央集権です。成否を分けるのはモデルの才能ではなく、それを取り囲む運用体系、すなわちハーネスだと信じているからです。すべての専門エージェントが中央の常時監督のもとで動くため、一つのフロアのミスが建物全体へ燃え広がることはありません。

## 核心アイデア: ハンドオフ連鎖ではなく一つのオーケストレーター

Glass Atrium は **Manager Pattern(中央集権マネージャーパターン)** を土台に据えています。メインの Claude Code セッションであるオーケストレーター一つが**戦略的制御プレーン**の役割を担い、ユーザーの意図をサブタスクへ*分解*し、専門エージェントへ*委任*し、その結果を*統合*します。自ら直接コードを書いたり成果物を作ったりすることは、決してありません。

対極のモデルである **Handoff Pattern(エージェント同士が制御権を直接やり取りする方式)** は、**あえてサポートしていません** — まさにそこが核心です。

Glass Atrium では、オーケストレーターが全体の文脈を一人で握ったまま、すべての委任境界で制御権を手放しません。

各サブタスクはキーワードではなく能力でルーティングされ(キーワードやエイリアスのマッチングは禁止)、誰が何を担うのかが正確に把握されます。各エージェントが構造化された完了記録を返すと、オーケストレーターが意図と結果の整合性を検証します。失敗したサブエージェントは失敗復旧ループ(リトライ → フォールバック → デバッガーエスカレーション)で処理され、スポーン深度・並行数の上限・エージェント別の予算が強制されるため、どんな委任も暴走することはありません。

この統制は手続きではなく**機械として働きます** — 規則をプロンプトに書き留めて、守られるのを願うだけでは終わりません。

ライフサイクルフックがすべてのツール呼び出しを横取りし、モニターがすべてのイベントを観測し、結果記録がすべての委任を追跡・監査します。フックに裏打ちされた強制、余すところない観測可能性、そして閉じた学習ループ — この組み合わせこそが、統制されないハンドオフシステムとの決定的な違いです。

## 動作の仕組み、最初から最後まで

一つのリクエストは、次のようにシステムを通り抜けていきます:

1. **あなたがオーケストレーター(メインセッション)に**何かを頼みます — バグ修正、機能企画、レポート作成。
2. **オーケストレーターが調査して分解**し、能力レジストリを参照しながら、専門エージェントのチームと実行順序を組み立てます。
3. **各サブタスクを委任**します — どの委任も、実際に実行する*前に* PreToolUse フック(危険コマンドの遮断・シークレットスキャン・範囲逸脱の表示・計画検証ゲート)を通ります。
4. **専門エージェントが作業を遂行**します — SubagentStart フックが注入した自身の範囲規則と、強制されたスポーン・ターン・ツール予算の枠内で。
5. **結果が記録されます。** 各エージェントが `[COMPLETION]` ブロックを吐き出すと、PostToolUse フックがこれを結果記録として捕捉します。
6. **オーケストレーターが統合**します — 検証済みの結果を一つの回答へまとめ、失敗が出れば復旧ループを回します。
7. **すべてが観測されます。** あらゆるコスト・結果・エージェントイベントが PostgreSQL と Atrium Monitor へリアルタイムにストリーミングされます。
8. **システムは自ら改善します。** バックグラウンドデーモンが結果記録と是正シグナルを読み取り、エージェントの指示を自動でパッチします — プロンプトを一行も触らずに済みます。

## 大きな仕事は慎重に、小さな仕事は素早く

このシステムは、**どの段階を経るかを作業の大きさに合わせて自ら決めます。** ユーザーが「慎重に段階を踏め」といちいち指示する必要はありません。

- **大きく複雑な仕事** — 計画を立て → 妥当かを点検し → 実際に作り → ちゃんと動くか試し → すべて済んだかを確かめて締めくくる、慎重な流れを自ら踏みます。
- **小さな仕事** — この手順を飛ばして手早く片づけます。

どちらへ進むかは、作業の大きさと複雑さを見てシステムが決めます。ワークフローとは、上記の段階を実行するやり方のことです — 「ワークフローで進めろ」と指定すれば、人間が一つずつ指示する代わりに、自動化エンジンが一気にまとめて処理します。踏むべき段階はどちらでも同じで、変わるのは実行のやり方だけです。

> 一行でまとめると: **何を経るかは作業の大きさが自動で決め、どう実行するかだけが選択肢です。**

## 内部の構成要素

- **専門エージェントの部隊** — キーワードではなく能力レジストリ(`agent-registry.json`)でルーティングされます(スタック別の開発者に、QA、企画・リサーチ・レポート、デザイン・オーディオ、セキュリティ、ウィキ、メタまで)。
- **階層化された規則システム** — グローバル憲章(`agents/GLASS_ATRIUM_GLOBAL_RULES.md`)、コアの横断規則(`rules/`)、範囲別の規則(`scoped/`)を、明示的なコンプライアンスマトリクスで束ねます。
- **ライフサイクルフックパイプライン** — すべてのツール境界でシークレット・危険コマンド・予算・結果記録を機械的に強制するフックスクリプト群(`hooks/`)。
- **自己改善ループ** — autoagent デーモン(`autoagent/`)が、たまった結果記録と是正シグナルをエージェント指示のパッチへと変え、安全なものだけを自動で適用します。適用前に原本を控えておき、問題が起きればそのまま元へ戻します。
- **Atrium Monitor** — Fastify 5 + Prisma 7 + React 18 ベースの 10 画面リアルタイムダッシュボード(`http://127.0.0.1:16145`)。
- **モデル・予算の直接割り当て** — モニターの **Models & budgets** 画面で、ドメイン別のモデルと呼び出しごとの USD ハードキャップを、設定ファイルを編集せずに割り当てます。
- **ライブアーキテクチャマップ** — モニターの System map 画面が、維持管理される 7 個の Mermaid ダイアグラムを、ライブのデーモン状態とともに描き出します。
- **ウィキ知識ストア** — 原本をキュレーションされたノートへ加工するパイプラインに、SQLite BM25 全文検索インデックスまで備えた LLM 専用ストア(`wiki/`)。リサーチエージェントの Web 調査結果がここに積み重なり、新しい調査・分析に取りかかる前にまず参照して、既存の知識を再利用します。
- **内部エージェントスキル** — エージェントとオーケストレーターが自動で呼び出す、漸進的開示方式の `SKILL.md` パッケージ([スキル](#スキル-内部の品質レイヤー)参照)。
- **ファイル単位のシンボリックリンクファームインストール** — ユーザー所有のファイルと衝突なく共存できるよう、ディレクトリではなくファイル単位で `~/.claude/<rel>` → `~/.glass-atrium/<rel>` のシンボリックリンクを冪等に生成します。
- **8 個のバックグラウンドデーモン** — `com.glass-atrium.*` の launchd ジョブ(keepalive 3 個: monitor・autoagent・wiki / スケジュール 5 個: ログローテーション・PG バックアップ・autoagent cycle・wiki compile・日次再起動)で、すべて `config.toml` に宣言されています。夜通し回る暖房・換気のように、誰かがスイッチを握っているのではなく、そもそも止まらないよう設計された建物の設備室です。
- **リポジトリにシークレット資料ゼロ** — peer-auth 専用の PostgreSQL(パスワードなし)、シークレットスキャンフック、リリースゲートの PII スキャン。

## スキル: 内部の品質レイヤー

Glass Atrium のスキルは、ユーザーが呼び出すコマンドでは**なく**、内部の品質・ガバナンスレイヤーです。

セッション開始時にグローバルへロードされ、Claude が目の前の作業に応じて自動的に有効化します。コードやデザインの規約、安全不変条件(「鉄則」)、運用・検証ゲート、ウィキ・Web ツールを扱い、一部はモニターのシグナル(System map のドリフトバッジ、Models & budgets の保存など)に反応して発動します。どのスキルが動いているかを気にする必要はありません — ハーネスが自ら選んで呼び出します。

## クイックスタート

**前提要件**: macOS · Claude Code CLI。残りはすべて、インストール TUI が同意を取ったうえで自動的に検出・インストールします。

```sh
curl -fsSL https://github.com/bettep-dev/glass-atrium/raw/main/install.sh | bash
```

対話メニューが開いたら **Install** を選んでください — インストールが終わると依存関係とデーモンが自動で設定・起動し、ダッシュボードには `http://127.0.0.1:16145` からアクセスできます。

> **インストールされたフォルダはそのままにしてください。** インストーラーはリリースバンドルをダウンロードして `~/.glass-atrium` フォルダへ展開し、**ファイル単位のシンボリックリンクファーム**(先に説明した方式)で Claude の設定ディレクトリと結びつけます。実体ファイルがここにあるので、フォルダを移動したり削除したりするとリンクが切れてしまいます。

### アンインストール

メニューから **Uninstall** を選んでください。インストールされたシンボリックリンクを取り除き、GA データベースをドロップして、既存の Claude システムから Atrium をきれいに切り離します。あなた自身のファイルには手を触れず、残留物も残しません。ただし、**データベースはバックアップなしで削除され**、再インストールすれば新しく作り直されます。

### Atrium Monitor のドキュメント作成方法

ドキュメント・レポート・要約・リファレンスの依頼は **glass-atrium-intel-reporter** へ、企画・要件定義・作業分解の依頼は **glass-atrium-intel-planner** へ委任されます。

どの形式で出力されるかは、**リクエストの文言が決めます**:

- デフォルトは、モニター上では**隠される「エージェント専用」のトークン最適化記録**です — 内容の形に応じて md・yaml・json・txt から選ぶ、あとで参照するための内部用の記録です。
- 「**HTML で**」「**Web ドキュメントとして**」「**チーム共有用に**」のように、共有や閲覧の意図をはっきり示すと、人が見て共有できる **HTML ドキュメント**が作られます(ダークテーマ・ダイアグラム・表を備えた単一ファイル)。

どちらの形式も、Atrium Monitor の**ドキュメント管理**画面に現れます。

ドキュメントをわざわざ依頼しなくても、記録しておく価値があると判断されれば、エージェント専用の記録が残ることがあります。

### 開発エージェントの追加

「○○をする開発エージェントを追加して」のように頼むと、オーケストレーターはすぐに新しいエージェントを作らず、まず**既存エージェントを拡張するか、新規に生成するか**を見極めます。**デフォルトは拡張であり、生成はゲートを通過したときだけ**許されます。

生成が許されるには、次の三つが**すべて独立して**満たされる必要があります:

- **異なる成果物の種類** — 作り出すファイルが構造的に区別される。
- **異なる決定ドメイン** — 正しい判断に必要な専門性が重ならない。
- **移転できない品質判断** — 一方の品質レビューを、もう一方の専門性だけでは代われない。

さらに、提案されたエージェントのドメインが**既存エージェントと一定比率以上重なると、生成はブロックされます** — 大きく重なるということは、新規作成ではなく拡張のシグナルだからです。

進行中には**ユーザー承認の一時停止が二度**あります。まずオーケストレーターが判断根拠と推奨案を示して生成か拡張かを確定し、続いてエージェントファイルとレジストリへ実際に書き込む直前に、もう一度確認します。

本文(システムプロンプト)は **glass-atrium-meta-prompt-engineer** が書き、登録は専用 CLI が受け持ちます。ユーザーは「追加して」と頼むだけでよく、拡張か新規かの判断と承認の要請はオーケストレーターが引き受けます。
