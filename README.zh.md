一次请求由编排器分解、委派并汇总给一支专业智能体团队。所有的成本、结果与智能体事件都会实时呈现在监控仪表盘上，而系统会从反复出现的失败中自行修正自己的指令——你无需编辑哪怕一行提示词。

> [!WARNING]
> 一经安装，后台守护进程便会**自动运行**。其中一部分每天会自动调用 Claude，因此即使不去管它也会消耗 token（产生费用）。
>
> 此外，这个框架是一个**高 token 消耗的系统**。由于编排器会把一次请求分解给一支专业智能体团队，所以一次请求会扩散为多次子智能体调用，token 消耗可能达到单个智能体的数倍。

## 是什么

Glass Atrium 是一个配置与工具层，能把 Claude Code CLI 变成一个受协调、可观测、能自我改进的多智能体系统。它**不是**一个新的智能体框架，也**不**取代 Claude Code——它安装到 `~/.claude/` 下，运行在你已经使用的 CLI 之上。

它在默认状态之上，作为一个统一一致的系统部署，增加了以下五样东西：

- 用一支**按能力进行路由的专业智能体队伍**（按技术栈划分的开发者、QA、企划、研究、设计、安全、wiki、元层）取代单一的通用提示词；
- 用一份矩阵来定义哪个智能体加载哪些规则的**分层规则系统**；
- 在每一个工具边界上*机械地*强制执行这些规则（密钥、危险命令、预算、产出物）的**生命周期钩子（hook）流水线**；
- 把所有成本、结果与智能体事件集中呈现于一处的**实时监控仪表盘**（Atrium Monitor）；
- 为修正反复出现的失败而重写智能体自身指令的**自我改进循环**。

## 为什么要做

单一的 `CLAUDE.md` 提示词无法扩展。一旦你要求一个模型既是 React 专家，*同时*又是数据库专家，*同时*又是安全审查员，*同时*又是企划者，那么指令之间就会彼此冲突，重要的规则会在庞大的文本堆里被稀释，而模型一旦负载过高，就会悄悄地开始偏离那些规则。

接下来明显会想到的一步——让智能体彼此移交任务——会把一个问题变成更糟的问题。在智能体之间的**移交（handoff）**系统中，控制权从智能体 A 转给 B，再转给 C，每个智能体都得自己强制执行自己的规则，可观测性在整条连锁中支离破碎，而中间任何一个出错的智能体都会在无人制衡的情况下传播下去。没有任何一方负起责任。

Glass Atrium 的诞生，是为了在不放弃掌控的前提下获得专业化的好处。所有专业智能体都在持续的中央监督下运行，所有重要规则都由框架而非提示词直接强制执行，系统会记录发生了什么，并——自动地——从中学习。

## 核心理念：不是移交连锁，而是单一编排器

Glass Atrium 建立在 **Manager Pattern（中心化的管理者模式）**之上。一个编排器——即主 Claude Code 会话——是永久的**战略控制平面**。它唯一的职责在于把你的意图*分解*为子任务，把每个任务*委派*给恰当的专业智能体，并*汇总*结果。它绝不亲自写代码或制造产出物。

与之相反的模型，即 **Handoff Pattern（对等的、智能体对智能体的控制权移交）**，**明确地不予支持**——而这正是关键所在。

在 Glass Atrium 中，编排器在每一个委派边界上都保持控制权：

- **它知道谁在做什么。**每个子任务都按能力而非关键字进行路由（禁止关键字/别名匹配）。
- **它验证每一个结果。**每个智能体都返回结构化的完成记录，由编排器核对意图与结果的一致性。
- **它可以中止、改派或上报。**失败的子智能体会触发失败恢复循环（重试 → 回退 → 上报给调试器）。
- **它有边界。**强制执行的派生（spawn）深度上限、并发上限以及每个智能体的预算，使任何委派都无法失控。

这种掌控**不仅是程序性的，更是机械性的。**它不止于把规则写在提示词里并指望被遵守——生命周期钩子会拦截每一次工具调用，监控会观测每一个事件，而结果记录使每一次委派都可追踪、可审计。正是这一组合——中央路由 **+** 由钩子支撑的强制执行 **+** 完整的可观测性 **+** 闭合的学习循环——构成了相较于失控移交系统的核心差异点。

## 运作方式，从头到尾

一次请求会这样穿越系统：

1. **你向编排器（主会话）**请求某件事——修复 bug、企划功能、撰写报告。
2. **编排器进行调查与分解**之后，参照能力注册表，组建一支专业智能体团队并确定执行顺序。
3. **它委派每一个子任务**——每一次委派都会在执行动作*之前*经过 **PreToolUse 钩子**（拦截危险命令、扫描密钥、标记越界范围、计划验证门）。
4. **专业智能体执行工作**——在 **SubagentStart 钩子**所注入的自身范围规则，以及强制执行的派生/轮次/工具预算之内。
5. **结果被记录。**每个智能体都会发出一个 `[COMPLETION]` 块，**PostToolUse 钩子**将其捕获为结果记录。
6. **编排器进行汇总**——把已验证的结果汇成单一答复，若失败则运行恢复循环。
7. **一切皆可观测。**所有成本、结果与智能体事件都会实时流向 PostgreSQL 和 Atrium Monitor。
8. **系统自行改进。**后台守护进程读取结果记录与纠正信号，自动给智能体指令打补丁——无需编辑哪怕一行提示词。

你不会从零件去拼装这个框架。在 `./glass-atrium` 的交互式菜单中选择一次 **Install**，整个系统就会在一次经过验证的运行中建立起来（配置、符号链接群、钩子接线、数据库、监控构建、健康门），安装后后台守护进程会自动运行。可执行的命令见[快速开始](#快速开始)。

## 大事审慎处理，小事快速处理

这个系统会**根据任务的大小自行决定要走哪些步骤。**你无需逐一指示「请审慎地一步步来」。

当任务又大又复杂时，系统会自动走一条审慎的流程。简单说，就是**先制定计划 → 检查该计划是否合理 → 实际构建 → 测试是否正常运作 → 确认是否完成并收尾**这样一个流程。反过来，小任务则会跳过这套程序而快速处理。至于走哪条路，系统会看任务的大小与复杂度自行判断。

这里可能会冒出「工作流（workflow）」这个词，它指的是**以何种方式来执行上述步骤**。如果你另行指定「以工作流方式进行」，那么同样的这些步骤就不再由人一个个去触发，而是由一个自动化引擎一次性打包处理。重要的是，检查、测试这类安全装置以及所经过的步骤，**无论走哪条路都是一样的**。变化的只是「如何执行（由自动引擎一次性处理，还是一步一步处理）」而已。

> 一句话概括：**要走哪些步骤由任务大小自动决定，而如何执行则只是可选项。**

## 内部构成要素

- **专业智能体队伍** — 按能力注册表（`agent-registry.json`）而非关键字进行路由（按技术栈划分的开发者、QA、企划/研究/报告、设计/音频、安全、wiki、元层）。
- **分层规则系统** — 用一份明确的合规矩阵将全局宪章（`agents/GLOBAL_RULES.md`）、核心横切规则（`rules/`）与按范围划分的规则（`scoped/`）绑在一起。
- **生命周期钩子流水线** — 在每一个工具边界上机械地强制执行密钥、危险命令、预算与结果记录的一组钩子脚本（`hooks/`）。
- **自我改进循环** — autoagent 守护进程（`autoagent/`）把累积的结果记录与纠正信号转化为智能体指令补丁，仅将安全的部分连同基于 git 的回滚一并自动应用。
- **Atrium Monitor** — 基于 Fastify 5 + Prisma 7 + React 18 的 10 个界面的实时仪表盘（`http://127.0.0.1:7842`）。
- **可编辑的模型 + 预算分配** — 监控的 **Models & budgets** 界面可在不修改配置文件的情况下，为各领域分配模型与每次调用的 USD 硬上限。
- **实时架构图** — 监控的 System map 界面将维护中的 7 张 Mermaid 图与实时守护进程状态一并渲染。
- **wiki 知识库** — 一个具备原始素材 → 经整理笔记的流水线，以及 SQLite BM25 全文检索索引的、仅供 LLM 使用的存储库（`wiki/`）。研究智能体的网络调查结果会积累于此，在开始新的调查/分析之前会先行查询，以复用既有知识。
- **内部智能体技能** — 由智能体和编排器自动调用的渐进式公开的 `SKILL.md` 包（参见[技能](#技能内部质量层)）。
- **按文件的符号链接群安装** — 为了能与用户自有文件共存，以文件而非目录为单位，幂等地创建 `~/.claude/<rel>` → `~/.glass-atrium/<rel>` 的符号链接。
- **8 个后台守护进程** — `com.glass-atrium.*` 的 launchd 作业（3 个 keepalive：monitor、autoagent、wiki；5 个定时：日志轮转、PG 备份、autoagent cycle、wiki compile、每日重启），全部在 `config.toml` 中声明。
- **存储库中零密钥素材** — 仅用 peer-auth 的 PostgreSQL（无密码）、密钥扫描钩子、发布门 PII 扫描。

## 技能：内部质量层

Glass Atrium 的技能**不是**由用户调用的命令——它们是在会话开始时全局加载、由 Claude 根据当下任务自动激活的内部质量与治理层。它们涵盖代码/设计规约、安全不变式（「铁律」）、运维与验证门、wiki 与 web 工具，其中一部分会响应监控信号（System map 漂移徽章、Models & budgets 保存等）而触发。**你无需知道哪个技能在运行**——框架会把它作为「就是能用」这件事的一部分挑选并调用。

## 快速开始

**前置要求**：macOS · Claude Code CLI。其余一切都由安装 TUI 征得同意后自动探测并安装。

```sh
curl -fsSL https://github.com/bettep-dev/glass-atrium/raw/main/install.sh | bash
```

交互式菜单打开后，请选择 **Install**——安装后依赖与守护进程会自动配置并运行，仪表盘会在 `http://127.0.0.1:7842` 处响应。

> 请把克隆下来的文件夹留在原处。Atrium 采用的是按文件的符号链接群（symlink farm），而非整个目录的替换：你的 Claude 配置目录中的文件会变成指向克隆所得的 `~/.glass-atrium` 内真实文件的符号链接，因此能与你自己的文件共存而不冲突。由于真实文件就存放在这个项目文件夹里，而安装好的符号链接都指向那里，所以一旦移动或删除该文件夹，这些链接就会失效。

### 卸载

在菜单中选择 **Uninstall**。它会移除已安装的符号链接并删除 GA 数据库，干净而彻底地把 Atrium 与你既有的 Claude 系统分离开来。你自己的文件不会受到任何改动，也不会残留任何 Atrium 的痕迹。数据库删除时不做备份——这是设计如此；重新安装时会重新创建一个全新的数据库。

### 如何撰写 Atrium Monitor 文档

请求文档、报告、摘要或参考资料时，会委派给 **intel-reporter**（负责报告、摘要、参考资料）；请求企划、需求定义或任务分解时，会委派给 **intel-planner**（负责企划、计划）。

会以何种格式输出，**由请求的措辞决定**：

- 默认值是监控中**隐藏的「仅智能体可见」的 token 优化记录**——会根据内容形态在 md、yaml、json、txt 中挑选。这是供日后再次参考的内部记录。
- 当你明确表达共享、阅览的意图，比如「**做成 HTML**」「**做成网页文档**」「**用于团队共享**」时，就会生成一份可供人阅读和共享的 **HTML 文档**（具备暗色主题、图表与表格的单一文件）。

两种格式都会出现在 Atrium Monitor 的**文档管理**界面中。

示例：

- 「把这次变更整理成一份回顾报告」→ 仅智能体可见的记录（默认值）。
- 「做成一份团队共享用的 HTML 报告」→ HTML 文档。

即便你没有明确请求文档，只要系统判断某内容值得留存，也可能会留下一份仅智能体可见的记录。

### 添加开发（DEV）智能体

当你像「添加一个做 OOO 的开发智能体」这样请求时，编排器不会立刻去创建一个新智能体，而是先判断**是要扩展（EXTEND）现有智能体，还是要新建（CREATE）**。**默认是扩展，而创建只有在通过门（gate）时**才被允许。

要允许创建，必须**全部独立地**满足以下三点：

- **不同的产出物类型** — 所生成的文件在结构上有所区别。
- **不同的决策领域** — 做出正确判断所需的专业性互不重叠。
- **不可转移的质量判断** — 一方的质量审查无法仅凭另一方的专业性来替代。

除此之外，如果所提议的智能体的领域与**现有智能体重叠达到一定比例以上，创建就会被拦截**——因为重叠很多并不是新建的信号，而是扩展现有智能体的信号（防止重复）。

进行过程中会有**两次用户审批的暂停**：

1. **创建 vs 扩展的决策审批** — 编排器提出判断依据与建议方案，由你来拍板。
2. **登记（提交）审批** — 在实际写入智能体文件与注册表之前，再确认一次。

正文（系统提示词）由 **meta-prompt-engineer** 编写，而登记本身由专用的 CLI 处理。概括来说，你只需请求「添加一个」即可——扩展与新建之间哪个合适，由编排器判断并提出建议，而要新建时则会征求你的批准。
