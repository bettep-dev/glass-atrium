**面向 Claude Code、由中心统一编排的自我改进型多智能体框架（harness）。**

由单个编排器（orchestrator）统一监督和掌控一支专业智能体队伍，绝不让智能体之间相互移交任务、最终陷入失控。

编排器接到一次请求后，将其分解并委派给一支专业智能体团队，再汇总各方结果。所有成本、结果与智能体事件都实时呈现在监控仪表盘上，如同隔着玻璃幕墙将内部尽收眼底；而系统还会从反复出现的失败中自行修正指令——你无需改动哪怕一行提示词。

> [!WARNING]
> 一经安装，后台守护进程便会**自动运行**。其中一部分每天都会自动调用 Claude，因此即便你不去管它，也会持续消耗 token（产生费用）。
>
> 此外，这套框架本身就是一个**高 token 消耗的系统**。编排器会把一次请求拆分给一支专业智能体团队，于是一次请求便会扩散成多次子智能体调用，token 消耗可能达到单个智能体的数倍。
>
> 不过，这些 token 并**非**通过直连 Anthropic API 按用量计费。本框架运行在你已经订阅的 **Claude Code CLI 之上**，因此无需单独的 API 密钥，也没有按量计费，而是直接使用该订阅套餐所含的 token。

## 是什么

Glass Atrium 是一个配置与工具层，能把 Claude Code CLI 变成一套受协调、可观测、可自我改进的多智能体系统。它**不是**新的智能体框架，也**不**取代 Claude Code——它安装在 `~/.claude/` 下，运行在你已经使用的 CLI 之上。

名字里的「玻璃中庭（Glass Atrium）」也是审视这套系统的一副镜片。与其把 AI 看作一个聪明的个体，不如把它想象成一栋 24 小时运转、即便深夜也内部通透可见的玻璃建筑——里面有众多专家各司其职，但所有工作都要经过唯一一位总管（编排器）。通透无遮的玻璃，加上所有路径最终都汇聚而至的中央总管——这两点正是名字所承载的身份。

在此之上，它还增添了以下五样东西——仅凭默认安装，它们便会绑成一套统一一致的系统：

- 一支**按能力进行路由的专业智能体队伍**（按技术栈划分的开发者，外加 QA、企划、研究、设计、安全、wiki、元层），取代单一的通用提示词；
- 一套**分层规则系统**，用一份矩阵定义哪个智能体加载哪些规则；
- 一条**生命周期钩子（hook）流水线**，在每一个工具边界上*机械地*强制执行这些规则（密钥、危险命令、预算、产出物）；
- 一块**实时监控仪表盘**（Atrium Monitor），把所有成本、结果与智能体事件集中呈现于一处；
- 一个**自我改进循环**，为修正反复出现的失败而重写智能体自身的指令。

## 为什么要做

单一的 `CLAUDE.md` 提示词无法扩展。一旦你要求同一个模型既当 React 专家，*同时*又当数据库专家，*同时*又当安全审查员，*同时*又当企划者，指令之间就会彼此冲突。真正重要的规则会淹没在庞大的文本里被稀释，而一旦负载过高，模型就会悄悄偏离那些规则。

所以工作必须拆分——与其依赖一位天才，不如让多位专家协作；一旦拆分，每位专家都能拥有各自干净的记忆空间（上下文），专注于自己的本职。

但顺理成章的下一步——让智能体彼此移交任务——只会把问题放得更大。在智能体之间的**移交（handoff）**系统里，控制权从智能体 A 交给 B，再交给 C。每个智能体都得自己执行自己的规则，可观测性沿着整条链条支离破碎，而中间任何一个出错的智能体都会在无人制衡的情况下把问题传播下去——始终没有谁负起最终责任。

Glass Atrium 正是为了在不放弃掌控的前提下享受专业化的红利而生——答案就是中心化。因为我们相信，决定成败的并非模型本身的天赋，而是环绕着它的运营体系，也就是这套框架（harness）。所有专业智能体都在中央的常态监督下运行，因此某一层的失误绝不会蔓延到整栋建筑。

## 核心理念：不是移交连锁，而是单一编排器

Glass Atrium 建立在 **Manager Pattern（中心化的管理者模式）** 之上。作为主 Claude Code 会话的单个编排器扮演着**战略控制平面**的角色，负责把你的意图*分解*为子任务，把每个任务*委派*给专业智能体，再*汇总*它们的结果。它绝不亲自写代码或产出交付物。

与之相对的模型——**Handoff Pattern（对等的、智能体对智能体的控制权移交）**——则**刻意不予支持**，而这恰恰是关键所在。

在 Glass Atrium 中，编排器独自掌握全局上下文，并在每一个委派边界上都牢牢握住控制权。

每个子任务都按能力而非关键字进行路由（禁止关键字/别名匹配），因此谁负责什么都被精确掌握。每个智能体都会返回一份结构化的完成记录，供编排器核对意图与结果是否一致。失败的子智能体交由失败恢复循环处理（重试 → 回退 → 上报给调试器），而强制生效的派生（spawn）深度、并发上限与各智能体预算，让任何一次委派都无法失控。互不重叠的独立子任务，即使你不主动要求，也会默认并行执行。

这种掌控并非停留在流程层面，而是**机械地运作**——它不止于把规则写进提示词、然后指望它们被遵守。

生命周期钩子拦截每一次工具调用，监控观测每一个事件，结果记录让每一次委派都可追踪、可审计。由钩子支撑的强制执行、完整的可观测性，再加上闭合的学习循环——正是这套组合，构成了它与失控移交系统之间的核心区别。

## 运作方式，从头到尾

一次请求会像这样在系统中穿行：

1. **你向编排器（主会话）**请求某件事——修复 bug、企划功能、撰写报告。
2. **编排器进行调查与分解**之后，参照能力注册表，组建一支专业智能体团队并确定执行顺序。
3. **它逐一委派子任务**——每一次委派都会在实际执行动作*之前*先经过 **PreToolUse 钩子**（拦截危险命令、扫描密钥、标记越界范围、计划验证门）。
4. **专业智能体执行工作**——在 **SubagentStart 钩子**所注入的自身范围规则，以及强制生效的派生/轮次/工具预算之内完成。较长的任务不会在预算耗尽时被强行中断，而是记录下进度、停在一个可恢复的状态。
5. **结果被记录。**每个智能体都会发出一个 `[COMPLETION]` 块，**PostToolUse 钩子**将其捕获为结果记录。
6. **编排器进行汇总**——把已验证的结果汇成单一答复；若出现失败，则运行恢复循环。
7. **一切皆可观测。**所有成本、结果与智能体事件都会实时流向 PostgreSQL 和 Atrium Monitor。
8. **系统自行改进。**后台守护进程读取结果记录与纠正信号，自动为智能体指令打补丁——无需改动哪怕一行提示词。

## 大事审慎处理，小事快速处理

这套系统会**根据任务的大小自行决定要走哪些步骤，**你无需逐一叮嘱「请审慎地一步步来」。

- **又大又复杂的任务** — 会自动走一条**先制定计划 → 检查计划是否合理 → 实际构建 → 测试能否正常运作 → 确认是否完成并收尾**的审慎流程。
- **小任务** — 跳过这套流程，快速处理。

至于走哪条路，由系统依据任务的大小与复杂度自行判断。而「工作流（workflow）」指的是执行上述步骤的方式——如果你指定「以工作流方式进行」，同样这些步骤便不再由人逐一触发，而是交给一个自动化引擎一次性打包处理。无论走哪条路，需要经过的步骤都一样，变的只是执行方式。

> 一句话概括：**要走哪些步骤由任务大小自动决定，而如何执行则只是可选项。**

## 内部构成要素

- **专业智能体队伍** — 按能力注册表（`agent-registry.json`）而非关键字进行路由（涵盖按技术栈划分的开发者、QA、企划/研究/报告、设计/音频、安全、wiki、元层）。
- **分层规则系统** — 用一份明确的合规矩阵，把全局宪章（`agents/GLASS_ATRIUM_GLOBAL_RULES.md`）、核心横切规则（`rules/`）与按范围划分的规则（`scoped/`）绑在一起。
- **生命周期钩子流水线** — 一组钩子脚本（`hooks/`），在每一个工具边界上机械地强制执行密钥、危险命令、预算与结果记录。
- **自我改进循环** — autoagent 守护进程（`autoagent/`）把累积的结果记录与纠正信号转化为智能体指令补丁，只有安全的部分才会自动应用。应用之前会先把原文另存一份，一旦出问题便原样回滚。除了这些指令补丁之外，每当开始新任务时，它还会把先前学到的成功与失败模式直接注入到该智能体的会话中，立即加以复用。
- **Atrium Monitor** — 基于 Fastify 5 + Prisma 7 + React 18、共 10 个界面的实时仪表盘（`http://127.0.0.1:16145`）。
- **模型与预算直接分配** — 在监控的 **Models & budgets** 界面，你无需修改配置文件即可为各领域分配模型，并设定每次调用的 USD 硬上限。
- **实时架构图** — 监控的 System map 界面会把维护中的 7 张 Mermaid 图与实时守护进程状态一并渲染。
- **wiki 知识库** — 一个仅供 LLM 使用的存储库（`wiki/`），配有「原始素材 → 经整理笔记」的加工流水线，以及 SQLite BM25 全文检索索引。研究智能体的网络调查结果会在此积累；每次开始新的调查/分析之前会先行查询，以复用既有知识。
- **内部智能体技能** — 由智能体和编排器自动调用、采用渐进式披露的 `SKILL.md` 包（参见[技能](#技能内部质量层)）。
- **按文件的符号链接群安装** — 为与用户自有文件互不冲突地共存，以文件（而非目录）为单位，幂等地创建 `~/.claude/<rel>` → `~/.glass-atrium/<rel>` 符号链接。
- **8 个后台守护进程** — 一批 `com.glass-atrium.*` 的 launchd 作业（3 个 keepalive：monitor、autoagent、wiki；5 个定时：日志轮转、PG 备份、autoagent cycle、wiki compile、每日重启），全部在 `config.toml` 中声明。就像整夜运转的供暖与通风，没有谁一直攥着开关，而是这栋建筑从设计之初就注定永不停歇的设备间。
- **存储库中零密钥素材** — 仅用 peer-auth 的 PostgreSQL（无密码）、密钥扫描钩子、发布门 PII 扫描。

## 技能：内部质量层

Glass Atrium 的技能**不是**由用户调用的命令，而是一个内部质量与治理层。

它们在会话开始时全局加载，由 Claude 根据当下的任务自动激活，涵盖代码/设计规约、安全不变式（「铁律」）、运维与验证门，以及 wiki 与 web 工具；其中一部分会响应监控信号（System map 漂移徽章、Models & budgets 保存等）而触发。你无需知道哪个技能正在运行——框架会自行挑选并调用。

## 快速开始

**前置要求**：macOS · Claude Code CLI。其余一切都由安装 TUI 征得同意后自动探测并安装。

```sh
curl -fsSL https://github.com/bettep-dev/glass-atrium/raw/main/install.sh | bash
```

交互式菜单打开后，请选择 **Install**——安装完成后，依赖与守护进程会自动配置并运行，仪表盘可在 `http://127.0.0.1:16145` 访问。

> **请把安装好的文件夹留在原处。** 安装程序会下载发布包、解压到 `~/.glass-atrium` 文件夹，再以**按文件的符号链接群**（前文所述的方式）与 Claude 配置目录相连。真实文件就存放在这里，因此一旦移动或删除该文件夹，这些链接便会失效。

### 卸载

在菜单中选择 **Uninstall**。它会移除已安装的符号链接并删除 GA 数据库，把 Atrium 从你既有的 Claude 系统中干净地剥离，既不改动任何用户自有文件，也不留下残余痕迹。但请注意，**数据库会先留下备份再删除**（转储保存在 `~/.claude/backups/postgres/`），重新安装时会创建一个新的数据库。备份不会自动恢复，如需旧数据，请用 `pg_restore` 手动恢复。

### 如何撰写 Atrium Monitor 文档

请求文档、报告、摘要或参考资料时，会委派给 **glass-atrium-intel-reporter**；请求企划、需求定义或任务分解时，则委派给 **glass-atrium-intel-planner**。

至于以何种格式输出，**由请求的措辞决定**：

- 默认是监控中**隐藏的「仅智能体可见」的 token 优化记录**——会根据内容形态在 md、yaml、json、txt 中挑选，是一份供日后再查的内部记录。
- 而当你明确表达共享、阅览的意图，比如「**做成 HTML**」「**做成网页文档**」「**用于团队共享**」，就会生成一份可供人阅读和共享的 **HTML 文档**（一个带暗色主题、图表与表格的单一文件）。

两种格式都会出现在 Atrium Monitor 的**文档管理**界面中，可在「进行中／已完成」状态之间切换，也可以用同一主题的新文档替换旧文档。

即便你没有专门请求文档，只要系统判断某段内容值得留存，也可能留下一份仅智能体可见的记录。

### 添加开发智能体

当你像「添加一个做 OOO 的开发智能体」这样请求时，编排器不会立刻去创建新智能体，而是先判断**该扩展现有智能体，还是新建一个**。**默认是扩展，创建只有在通过门禁时**才被允许。

要允许创建，必须**全部独立地**满足以下三点：

- **不同的产出物类型** — 所生成的文件在结构上彼此有别。
- **不同的决策领域** — 做出正确判断所需的专业能力互不重叠。
- **不可转移的质量判断** — 一方的质量审查无法仅凭另一方的专业能力来替代。

除此之外，如果所提议智能体的领域与**现有智能体重叠达到一定比例以上，创建就会被拦截**——因为大量重叠并不是新建的信号，而是扩展现有智能体的信号。

整个过程中会有**两次征求用户批准的暂停**：先由编排器给出判断依据与建议方案，敲定是创建还是扩展；随后在真正写入智能体文件与注册表之前，再确认一次。

正文（系统提示词）由 **glass-atrium-meta-prompt-engineer** 撰写，而登记本身交给专用的 CLI 处理。概括来说，你只需请求「添加一个」即可——扩展还是新建哪个更合适，由编排器判断并给出建议；确需新建时，则会征求你的批准。
