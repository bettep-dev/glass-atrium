**Claude Code를 위한, 중앙에서 오케스트레이션되는 자기개선형 멀티 에이전트 하니스.**

오케스트레이터가 요청 하나를 받아 전문 에이전트 팀으로 분해해 위임하고, 그 결과를 종합합니다. 모든 비용과 결과, 에이전트 이벤트는 유리 벽 너머를 들여다보듯 실시간 모니터 대시보드에 드러나며, 시스템은 반복되는 실패로부터 스스로 지침을 고쳐 나갑니다 — 프롬프트를 손볼 필요가 없습니다.

> [!WARNING]
> 설치하면 백그라운드 데몬이 **자동으로 실행**됩니다. 이 중 일부는 매일 자동으로 Claude를 호출하므로, 그대로 두어도 토큰(비용)이 소비됩니다.
>
> 또한 이 하니스는 **토큰을 많이 쓰는 시스템**입니다. 오케스트레이터가 요청 하나를 전문 에이전트 팀으로 쪼개므로, 요청 하나가 여러 서브에이전트 호출로 번져 토큰 소비가 단일 에이전트의 몇 배에 이를 수 있습니다.
>
> 다만 이 토큰은 Anthropic API에 직접 연결해 사용량만큼 청구되는 방식이 아닙니다. 이 하니스는 이미 구독 중인 **Claude Code CLI 위에서** 동작하므로, 별도의 API 키나 종량 과금 없이 그 구독 요금제에 포함된 토큰을 그대로 쓰는 구조입니다.

## 무엇인가

Glass Atrium은 Claude Code CLI를 조율되고 관측 가능하며 스스로 개선되는 멀티 에이전트 시스템으로 바꿔 주는 설정·도구 계층입니다. 새로운 에이전트 프레임워크가 **아니고** Claude Code를 대체하지도 **않습니다** — `~/.claude/`에 설치되어 이미 쓰고 있는 CLI 위에서 동작합니다.

이름의 '유리 아트리움'은 이 시스템을 바라보는 렌즈이기도 합니다. AI를 똑똑한 한 사람으로 여기는 대신, 어두운 밤에도 속이 훤히 비치는 채로 24시간 돌아가는 유리 건물 하나 — 그 안에서 여러 전문가가 일하되 모든 업무가 총괄 한 명을 거치는 회사로 보는 것입니다. 숨김 없는 유리, 그리고 모든 길이 모이는 중앙의 총괄 — 이 둘이 이름에 담긴 정체성입니다.

여기에 더해, 기본 설치만으로 하나의 일관된 시스템으로 묶이는 다섯 가지를 보탭니다:

- 범용 프롬프트 하나로 모든 걸 처리하는 대신, **역량을 기준으로 라우팅하는 전문 에이전트 부대** (개발자는 스택별로, 그 밖에 QA·기획·리서치·디자인·보안·위키·메타);
- 어느 에이전트가 어떤 규칙을 불러올지 매트릭스로 정의하는 **계층화된 규칙 시스템**;
- 모든 도구 경계에서 그 규칙(시크릿·위험 명령·예산·결과물)을 *기계적으로* 강제하는 **라이프사이클 훅 파이프라인**;
- 모든 비용과 결과, 에이전트 이벤트를 한곳에 모아 보여 주는 **실시간 모니터링 대시보드** (Atrium Monitor);
- 반복되는 실패를 바로잡으려고 에이전트 스스로 자기 지침을 다시 쓰는 **자기개선 루프**.

## 왜 만들었나

`CLAUDE.md` 프롬프트 하나로는 규모를 감당할 수 없습니다. 한 모델에게 React 전문가이면서 *동시에* 데이터베이스 전문가, *동시에* 보안 리뷰어, *동시에* 기획자가 되라고 요구하는 순간 지침끼리 충돌합니다. 정작 중요한 규칙은 방대한 텍스트에 묻혀 희석되고, 부하가 걸리면 모델은 슬그머니 그 규칙을 벗어납니다.

그래서 일은 나눠야 합니다 — 한 명의 천재보다 여러 전문가의 협업이 낫고, 나누면 각자 깨끗한 기억 공간(컨텍스트)을 갖고 자기 일에 집중할 수 있습니다.

그렇다고 다음 수순 — 에이전트끼리 작업을 넘기는 것 — 은 문제를 더 키웁니다. 에이전트 간 **핸드오프** 시스템에서는 제어권이 A에서 B로, B에서 다시 C로 이동합니다. 각 에이전트가 자기 규칙을 알아서 지켜야 하고 관측 가능성은 흩어지며, 오작동 에이전트 하나가 견제 없이 전파돼도 끝까지 책임지는 주체가 없습니다.

Glass Atrium은 통제를 놓지 않으면서도 전문화의 이점을 누리려고 만들었습니다 — 답은 중앙 집중입니다. 성패를 가르는 것은 모델의 재능이 아니라 그를 둘러싼 운영 체계, 즉 하니스라고 믿기 때문입니다. 모든 전문 에이전트가 중앙의 상시 감독 아래 실행되므로, 한 층의 실수가 건물 전체로 번지지 않습니다.

## 핵심 아이디어: 핸드오프 연쇄가 아닌 하나의 오케스트레이터

Glass Atrium은 **Manager Pattern(중앙집중 매니저 패턴)** 을 토대로 삼습니다. 메인 Claude Code 세션인 오케스트레이터 하나가 **전략적 제어 평면** 역할을 맡아, 사용자의 의도를 하위 작업으로 *분해*하고 전문 에이전트에게 *위임*하며 그 결과를 *종합*합니다. 직접 코드를 쓰거나 결과물을 만드는 일은 결코 없습니다.

반대편 모델인 **Handoff Pattern(에이전트끼리 제어권을 직접 주고받는 방식)** 은 **의도적으로 지원하지 않습니다** — 바로 그 점이 핵심입니다.

Glass Atrium에서 오케스트레이터는 전체 맥락을 혼자 쥔 채, 모든 위임 경계에서 제어권을 유지합니다.

각 하위 작업은 키워드가 아니라 역량으로 라우팅되어(키워드·별칭 매칭 금지) 누가 무엇을 하는지 정확히 파악됩니다. 각 에이전트가 구조화된 완료 기록을 돌려주면, 오케스트레이터가 의도와 결과가 일치하는지 검증합니다. 실패한 서브에이전트는 실패 복구 루프(재시도 → 폴백 → 디버거 에스컬레이션)로 처리되며, 스폰 깊이·동시성 상한·에이전트별 예산이 강제되어 어떤 위임도 폭주하지 않습니다.

이 통제는 절차가 아니라 **기계적으로 작동합니다** — 규칙을 프롬프트에 적어 두고 지켜지길 바라는 데서 멈추지 않습니다.

라이프사이클 훅이 모든 도구 호출을 가로채고, 모니터가 모든 이벤트를 관측하며, 결과 기록이 모든 위임을 추적·감사합니다. 훅으로 뒷받침되는 강제와 완전한 관측 가능성, 닫힌 학습 루프 — 이 조합이 통제되지 않는 핸드오프 시스템과의 핵심 차이입니다.

## 동작 방식, 처음부터 끝까지

하나의 요청은 다음과 같이 시스템을 거쳐 갑니다:

1. **사용자가 오케스트레이터(메인 세션)에게** 무언가를 요청합니다 — 버그 수정, 기능 기획, 보고서 작성.
2. **오케스트레이터가 조사하고 분해한** 뒤, 역량 레지스트리를 참조해 전문 에이전트 팀과 실행 순서를 짭니다.
3. **각 하위 작업을 위임합니다** — 모든 위임은 실제 실행 *전에* PreToolUse 훅(위험 명령 차단·시크릿 스캔·범위 이탈 표시·계획 검증 게이트)을 통과합니다.
4. **전문 에이전트가 작업을 수행합니다** — SubagentStart 훅이 주입한 자기 범위 규칙과 강제된 스폰·턴·도구 예산 안에서.
5. **결과가 기록됩니다.** 각 에이전트가 `[COMPLETION]` 블록을 내보내면, PostToolUse 훅이 이를 결과 기록으로 포착합니다.
6. **오케스트레이터가 종합합니다** — 검증된 결과를 하나의 답변으로 묶고, 실패가 나면 복구 루프를 돌립니다.
7. **모든 것이 관측됩니다.** 모든 비용과 결과, 에이전트 이벤트가 PostgreSQL과 Atrium Monitor로 실시간 스트리밍됩니다.
8. **시스템은 스스로 개선됩니다.** 백그라운드 데몬이 결과 기록과 교정 신호를 읽어 에이전트 지침을 자동으로 패치합니다 — 프롬프트를 한 줄도 손대지 않아도 됩니다.

## 큰 일은 신중하게, 작은 일은 빠르게

이 시스템은 **어떤 단계를 거칠지 작업 크기에 맞춰 스스로 정합니다.** 사용자가 "신중하게 단계를 밟아라"라고 일일이 일러 줄 필요가 없습니다.

- **크고 복잡한 일** — 계획을 세우고 → 타당한지 점검하고 → 만들고 → 동작을 시험하고 → 다 됐는지 확인해 마무리하는 신중한 흐름을 알아서 밟습니다.
- **작은 일** — 이 절차를 건너뛰고 빠르게 처리합니다.

어느 쪽으로 갈지는 작업의 크기와 복잡도로 시스템이 정합니다. 워크플로우는 위 단계들을 실행하는 방식을 말합니다 — "워크플로우로 진행하라"고 지정하면, 사람이 하나씩 시키는 대신 자동화 엔진이 한 번에 묶어서 처리합니다. 거쳐야 할 단계는 어느 쪽이든 같고, 달라지는 건 실행 방식뿐입니다.

> 한 줄 요약: **무엇을 거치느냐는 작업 크기가 자동으로 정하고, 어떻게 실행하느냐만 선택 사항입니다.**

## 내부 구성 요소

- **전문 에이전트 부대** — 키워드가 아니라 역량 레지스트리(`agent-registry.json`)로 라우팅됩니다(스택별 개발자에 QA, 기획·리서치·리포트, 디자인·오디오, 보안, 위키, 메타까지).
- **계층화된 규칙 시스템** — 전역 헌장(`agents/GLASS_ATRIUM_GLOBAL_RULES.md`), 핵심 횡단 규칙(`rules/`), 범위별 규칙(`scoped/`)을 명시적인 컴플라이언스 매트릭스로 묶습니다.
- **라이프사이클 훅 파이프라인** — 모든 도구 경계에서 시크릿·위험 명령·예산·결과 기록을 기계적으로 강제하는 훅 스크립트 모음(`hooks/`).
- **자기개선 루프** — autoagent 데몬(`autoagent/`)이 쌓인 결과 기록과 교정 신호를 에이전트 지침 패치로 바꾸고, 안전한 것만 자동 적용합니다. 적용 전 원본을 보관해 두었다가 문제가 생기면 되돌립니다.
- **Atrium Monitor** — Fastify 5 + Prisma 7 + React 18 기반의 10개 화면 실시간 대시보드(`http://127.0.0.1:16145`).
- **모델·예산 직접 할당** — 모니터의 **Models & budgets** 화면에서 설정 파일을 건드리지 않고도 도메인별 모델과 호출당 USD 하드 캡을 할당합니다.
- **라이브 아키텍처 맵** — 모니터의 System map 화면이 관리 중인 7개 Mermaid 다이어그램을 실시간 데몬 상태와 함께 렌더링합니다.
- **위키 지식 저장소** — 원본을 큐레이션된 노트로 가공하는 파이프라인에 SQLite BM25 전문 검색 인덱스까지 갖춘 LLM 전용 저장소(`wiki/`). 리서치 에이전트의 웹 조사 결과가 이곳에 쌓이고, 새 조사·분석 전에 먼저 조회해 기존 지식을 재사용합니다.
- **내부 에이전트 스킬** — 에이전트와 오케스트레이터가 자동으로 호출하는 점진적 공개 방식의 `SKILL.md` 패키지([스킬](#스킬-내부-품질-계층) 참고).
- **파일 단위 심링크 팜 설치** — 사용자 소유 파일과 충돌 없이 공존하도록, 디렉터리가 아닌 파일 단위로 `~/.claude/<rel>` → `~/.glass-atrium/<rel>` 심링크를 멱등하게 생성합니다.
- **8개 백그라운드 데몬** — `com.glass-atrium.*` launchd 작업(keepalive 3개: monitor·autoagent·wiki / 스케줄 5개: 로그 회전·PG 백업·autoagent cycle·wiki compile·일일 재시작)으로, 모두 `config.toml` 에 선언되어 있습니다. 밤새 도는 난방·환기처럼, 누가 스위치를 쥔 게 아니라 애초에 멈추지 않게 설계된 건물의 설비실입니다.
- **저장소에 시크릿 자료 제로** — peer-auth 전용 PostgreSQL(비밀번호 없음), 시크릿 스캔 훅, 릴리스 게이트 PII 스캔.

## 스킬: 내부 품질 계층

Glass Atrium의 스킬은 사용자가 호출하는 명령이 **아니라**, 내부 품질·거버넌스 계층입니다.

세션 시작 시 전역으로 로드되고, Claude가 작업에 맞춰 알아서 활성화합니다. 코드·디자인 규약, 안전 불변식("철칙"), 운영·검증 게이트, 위키·웹 도구를 다루며, 일부는 모니터 신호(System map 드리프트 배지, Models & budgets 저장 등)에 반응해 발동합니다. 어느 스킬이 도는지 알 필요는 없습니다 — 하니스가 알아서 골라 호출합니다.

## 빠른 시작

**사전 요구사항**: macOS · Claude Code CLI. 나머지는 모두 설치 TUI가 동의를 받아 자동으로 탐지하고 설치합니다.

```sh
curl -fsSL https://github.com/bettep-dev/glass-atrium/raw/main/install.sh | bash
```

대화형 메뉴가 열리면 **Install** 을 선택하세요 — 설치가 끝나면 의존성과 데몬이 자동으로 설정·실행되고, 대시보드는 `http://127.0.0.1:16145` 에서 접근할 수 있습니다.

> **설치된 폴더는 그대로 두세요.** 설치 프로그램은 릴리스 번들을 내려받아 `~/.glass-atrium` 폴더에 풀어 놓고, **파일 단위 심링크 팜**(앞서 설명한 방식)으로 Claude 설정 디렉터리와 연결합니다. 실제 파일이 여기 있으므로, 폴더를 옮기거나 지우면 링크가 끊어집니다.

### 제거

메뉴에서 **Uninstall** 을 선택하세요. 설치된 심링크를 제거하고 GA 데이터베이스를 드롭해 기존 Claude 시스템에서 아트리움을 깨끗이 떼어냅니다. 사용자 소유 파일은 그대로 두고, 잔여물도 남기지 않습니다. 단, **데이터베이스는 백업을 남긴 뒤에 삭제되며**(덤프는 `~/.claude/backups/postgres/` 에 보관), 재설치하면 새로 만들어집니다. 백업이 자동으로 복원되지는 않으므로, 이전 데이터가 필요하면 `pg_restore` 로 직접 복원하세요.

### 아트리움 모니터 문서 작성 방법

문서·보고서·요약·레퍼런스 요청은 **glass-atrium-intel-reporter**에게, 기획·요구사항 정의·작업 분해 요청은 **glass-atrium-intel-planner**에게 위임됩니다.

어떤 형식으로 나올지는 **요청 문구가 결정합니다**:

- 기본값은 모니터에서 **숨겨지는 "에이전트 전용" 토큰 최적화 기록**입니다 — 내용의 형태에 따라 md·yaml·json·txt 중에서 고르며, 나중에 참고할 내부용 기록입니다.
- "**HTML로**", "**웹 문서로**", "**팀 공유용으로**" 처럼 공유하거나 열람할 의도를 분명히 밝히면, 사람이 보고 공유할 수 있는 **HTML 문서**가 만들어집니다(다크 테마와 다이어그램, 표가 있는 단일 파일).

두 형식 모두 Atrium Monitor의 **문서 관리** 화면에 나타납니다.

문서를 따로 요청하지 않더라도, 기록해 둘 만하다고 판단되면 에이전트 전용 기록이 남을 수 있습니다.

### 개발 에이전트 추가

"OOO 하는 개발 에이전트를 추가해줘" 처럼 요청하면, 오케스트레이터는 곧장 새 에이전트를 만들지 않고 먼저 **기존 에이전트를 확장할지 새로 생성할지** 를 판단합니다. **기본은 확장이고, 생성은 게이트를 통과할 때만** 허용됩니다.

생성이 허용되려면 세 가지가 **모두 독립적으로** 충족되어야 합니다:

- **다른 산출물 유형** — 만들어 내는 파일이 구조적으로 구별됩니다.
- **다른 결정 도메인** — 올바른 판단에 필요한 전문성이 겹치지 않습니다.
- **이전 불가능한 품질 판단** — 한쪽의 품질 검토를 다른 쪽 전문성만으로는 대신할 수 없습니다.

여기에 더해, 제안된 에이전트의 도메인이 **기존 에이전트와 일정 비율 이상 겹치면 생성이 차단**됩니다 — 많이 겹친다는 건 확장 신호이지 신규 생성 신호가 아니기 때문입니다.

진행 중에는 **사용자 승인 일시정지가 두 번** 있습니다. 먼저 오케스트레이터가 판단 근거와 권고안을 제시해 생성 vs 확장을 확정하고, 그다음 에이전트 파일과 레지스트리에 실제로 기록하기 직전 한 번 더 확인합니다.

본문(시스템 프롬프트)은 **glass-atrium-meta-prompt-engineer**가 작성하고, 등록은 전용 CLI가 처리합니다. 사용자는 "추가해줘"라고 요청하기만 하면 되고, 확장·신규 판단과 승인 요청은 오케스트레이터가 맡습니다.
